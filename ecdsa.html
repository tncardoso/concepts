Section 1

What is a Digital Signature?

A digital signature is the cryptographic equivalent of a handwritten signature, but far more powerful. It provides three critical guarantees:

🔐

Authentication

Proves the signature was created by someone with the private key.

🛡️

Integrity

Any change to the message invalidates the signature.

🚫

Non-repudiation

The signer cannot deny having signed the message.

The two-key model

Digital signatures use asymmetric cryptography: you get a pair of keys. The private key is kept secret and used to sign. The public key is shared with the world and used to verify. The magic: anyone can verify, but only the private key holder can sign.

ECDSA is one specific construction of this model using elliptic curves. To understand it, we need to start with the math.

Section 2

Elliptic Curves

An elliptic curve over the real numbers is defined by the equation:

$$y^2 = x^3 + ax + b$$

where $a$ and $b$ are constants. For cryptography, we need the curve to be non-singular, meaning $4a^3 + 27b^2 \neq 0$ (no self-intersections or cusps).

Adjust the parameters below to see how the curve changes. Notice the characteristic symmetry about the x-axis — for every point $(x, y)$ on the curve, $(x, -y)$ is also on the curve.

⚠️ Real numbers vs. finite fields: The visualizations above use real numbers ($\mathbb{R}$) for intuition. Real ECDSA operates over a finite field $\mathbb{F}_p$ (integers mod a large prime $p$). The same equations hold, but points wrap around modularly — think of them as living on a torus rather than an infinite plane. The geometric rules (addition, doubling) still apply algebraically.

Section 3

Point Addition

The key operation on an elliptic curve is point addition. Given two points $P$ and $Q$ on the curve, we can find a third point $P + Q$ that is also on the curve. This is defined geometrically:

1

Draw a line through P and Q

This line will intersect the curve at a third point (by Bézout's theorem).

2

Reflect over the x-axis

The reflection of that third intersection point is $P + Q$.

Click two points on the curve below to see the addition in action. The purple line connects $P$ and $Q$ and hits a third point. The green point (reflection) is the result $P + Q$.

Point Doubling

When $P = Q$, we can't draw a line through two distinct points. Instead, we draw the tangent line at $P$ and reflect its intersection with the curve. This gives $2P = P + P$. Click the button to see doubling:

Algebraic Formula

For points $P = (x_1, y_1)$ and $Q = (x_2, y_2)$ on $y^2 = x^3 + ax + b$:

$$\lambda = \begin{cases} \frac{y_2 - y_1}{x_2 - x_1} & \text{if } P \neq Q \\[6pt] \frac{3x_1^2 + a}{2y_1} & \text{if } P = Q \end{cases}$$ $$x_3 = \lambda^2 - x_1 - x_2, \qquad y_3 = \lambda(x_1 - x_3) - y_1$$

The identity element is a special "point at infinity" $\mathcal{O}$, acting like zero: $P + \mathcal{O} = P$. And $P + (-P) = \mathcal{O}$, where $-P = (x, -y)$.

Why does the geometric rule work?

A line $y = \lambda x + c$ and the curve $y^2 = x^3 + ax + b$ intersect when:

$$(\lambda x + c)^2 = x^3 + ax + b \implies x^3 - \lambda^2 x^2 + (a - 2\lambda c)x + (b - c^2) = 0$$

This is a cubic in $x$ with three roots (counting multiplicity) corresponding to the x-coordinates of $P$, $Q$, and the third intersection point $R' = (x_3, -y_3)$. By Vieta's formulas: $x_1 + x_2 + x_3' = \lambda^2$, so $x_3 = \lambda^2 - x_1 - x_2$.

Section 4

Scalar Multiplication

If we can add points, we can also multiply by a scalar: $nP = \underbrace{P + P + \cdots + P}_{n \text{ times}}$. This is called scalar multiplication.

The interactive below computes $nP$ step by step for a point $P$ on the curve. Watch how the result "jumps around" the curve, seemingly at random:

Scalar Multiplication: compute $nP$

This "random jumping" is crucial. After just a few additions, the relationship between $n$ and $nP$ looks completely chaotic — even though each step is deterministic. This is the foundation of the security.

Double-and-Add: efficient computation

Computing $nP$ naively requires $n$ additions. But using the double-and-add method (analogous to exponentiation by squaring), we only need $O(\log n)$ operations:

Express $n$ in binary. For each bit from MSB to LSB: double the accumulator, and if the bit is 1, add $P$. For example, $21P = 2(2(2(2(2(P) + P) + P) + P) + P)$.

This means computing $nP$ for a 256-bit $n$ takes at most 510 point operations — fast!

Section 5

The Trap Door: Elliptic Curve Discrete Logarithm Problem

Here's the crux of elliptic curve cryptography:

The One-Way Function

Given a base point $G$ and a scalar $d$, computing $Q = dG$ is easy (polynomial time).

Given $G$ and $Q$, finding $d$ such that $Q = dG$ is the Elliptic Curve Discrete Logarithm Problem (ECDLP) — believed to be hard (no known sub-exponential algorithm for well-chosen curves).

✓ Easy: d, G → dG

Scalar multiplication: $O(\log d)$ steps via double-and-add. Milliseconds even for 256-bit numbers.

✗ Hard: G, dG → d

Discrete logarithm: best known classical algorithm is $O(\sqrt{n})$ — for secp256k1 this is ~$2^{128}$ operations. Infeasible.

To build intuition for why this is hard, the scatter plot below shows the mapping $n \mapsto nG$ for points on a small finite field curve. Each dot represents $nP$ for $n = 1, 2, \ldots$ — see how the x-coordinates jump around unpredictably:

Visualization: $nG$ appears random

Each cyan dot is the x-coordinate of $nP$ (ordered left-to-right by $n$). The sequence appears completely random — there's no visible pattern to predict $nP$ from $n$ or to recover $n$ from $nP$. This pseudorandomness is what makes the ECDLP hard.

Comparison with RSA

RSA relies on the hardness of integer factorization: $n = pq$ where $p, q$ are large primes. The best known algorithm (General Number Field Sieve) runs in sub-exponential time: $L_n[1/3] \approx \exp\!\bigl((64/9)^{1/3} (\ln n)^{1/3}(\ln \ln n)^{2/3}\bigr)$.

ECDLP's best known classical algorithm (Pollard's rho) runs in $O(\sqrt{n})$ — fully exponential. This means elliptic curves provide more security per bit:

  • 256-bit ECC ≈ 3072-bit RSA security level
  • 384-bit ECC ≈ 7680-bit RSA security level

Smaller keys mean faster operations, less bandwidth, and less energy — critical for embedded devices and blockchains.

Section 6

Key Generation

Now we have all the building blocks. Let's generate an ECDSA key pair:

1

Choose a curve and base point

Everyone agrees on a standard curve (e.g., secp256k1 for Bitcoin, P-256 for TLS). The curve defines the field $\mathbb{F}_p$, parameters $a, b$, and a base point $G$ of large prime order $n$ (meaning $nG = \mathcal{O}$).

2

Pick a random private key $d$

Choose a random integer $d \in \{1, 2, \ldots, n-1\}$. This is your private key. Keep it secret!

3

Compute the public key $Q = dG$

Scalar-multiply the base point by your private key. The result $Q$ is your public key. Share it with the world!

The one-way property guarantees: anyone can compute $Q$ from $d$, but nobody can recover $d$ from $Q$.

Section 7

Signing a Message

To sign a message $m$, the signer with private key $d$ does the following:

1

Hash the message

Compute $e = H(m)$ where $H$ is a cryptographic hash (e.g., SHA-256). Treat $e$ as an integer.

2

Generate a random nonce $k$

Pick a random $k \in \{1, \ldots, n-1\}$. This must be truly random and never reused — reusing $k$ leaks the private key!

3

Compute the curve point

Calculate $(x_1, y_1) = kG$. Let $r = x_1 \bmod n$. If $r = 0$, pick a new $k$.

4

Compute $s$

$s = k^{-1}(e + d \cdot r) \bmod n$. If $s = 0$, pick a new $k$.

ECDSA Signature

The signature is the pair $(r, s)$. Both values are integers mod $n$ (256-bit for secp256k1).

⚠️ Why nonce reuse is catastrophic: If you sign two messages $m_1, m_2$ with the same $k$, you get $(r, s_1)$ and $(r, s_2)$ where $s_i = k^{-1}(H(m_i) + dr)$. Then $s_1 - s_2 = k^{-1}(H(m_1) - H(m_2))$, so $k = \frac{H(m_1) - H(m_2)}{s_1 - s_2}$ and $d = \frac{s_1 k - H(m_1)}{r}$. This is exactly how the PlayStation 3 was hacked in 2010!

Section 8

Verifying a Signature

Given the message $m$, signature $(r, s)$, and public key $Q$, a verifier checks:

1

Hash the message

Compute $e = H(m)$, same as the signer.

2

Compute $w = s^{-1} \bmod n$

The modular inverse of $s$.

3

Compute $u_1 = e \cdot w \bmod n$ and $u_2 = r \cdot w \bmod n$

These are the "weights" for the base point and public key.

4

Compute $(x_2, y_2) = u_1 G + u_2 Q$

A linear combination of two scalar multiplications.

5

Accept if $x_2 \bmod n = r$

The signature is valid if and only if this holds.

Why does verification work? (The math)

During signing: $s = k^{-1}(e + dr) \bmod n$, which means $k = s^{-1}(e + dr) \bmod n$.

During verification, the verifier computes:

$$u_1 G + u_2 Q = ewG + rwQ = ewG + rwdG = (e + dr)wG = (e+dr)s^{-1}G$$

Since $k = s^{-1}(e + dr) \bmod n$, this equals $kG$. The x-coordinate of $kG$ is exactly $r$ (from step 3 of signing). So the check $x_2 \equiv r \pmod{n}$ passes! ✓

Conversely, a forged signature would require solving for $s$ such that $u_1 G + u_2 Q$ yields the right x-coordinate — without knowing $d$ — which requires solving the discrete logarithm problem.

Section 10

Why Is ECDSA Secure?

ECDSA's security rests on several pillars:

🔒

ECDLP Hardness

Recovering $d$ from $Q = dG$ requires ~$\sqrt{n}$ operations. For secp256k1, that's $2^{128}$ — more than the number of atoms in the observable universe squared.

🎲

Random Nonce

Each signature uses a fresh random $k$. This ensures signatures on the same message look different, preventing statistical attacks.

🔗

Hash Function

SHA-256 is preimage-resistant: given a hash output, finding an input is infeasible. This prevents existential forgery.

📐

Curve Choice

Standard curves are carefully chosen to resist known attacks: twist attacks, small subgroup attacks, invalid curve attacks, etc.

Attack costs

For secp256k1 (the Bitcoin curve, order $n \approx 2^{256}$):

Brute force

Try all $d \in \{1, \ldots, n-1\}$: $O(2^{256})$ operations. Completely infeasible.

Pollard's rho (best classical)

Birthday-paradox-based: $O(\sqrt{n}) = O(2^{128})$. Still infeasible (~$10^{28}$ years on current hardware).

Shor's algorithm (quantum)

Solves ECDLP in $O((\log n)^3)$ on a large quantum computer. This is why post-quantum cryptography is being standardized.
